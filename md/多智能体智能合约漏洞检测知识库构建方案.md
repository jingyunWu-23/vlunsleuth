# 多智能体智能合约漏洞检测系统知识库构建与处理方案

## 1. 项目背景

当前系统拟构建一个面向智能合约安全检测的多智能体框架，主要包含以下检测与处理模块：

- **ChainSafeFort**：由四个 LSTM 二分类模型组成，分别检测四种已知漏洞；
- **GCN 跨合约检测模块**：用于检测跨合约已知漏洞，并提供粗粒度漏洞位置或调用关系证据；
- **DeepSVNN**：用于检测未知漏洞或异常合约行为，但不能直接确定漏洞类型；
- **Function Scoring Agent**：对函数进行风险评分，筛选高风险函数；
- **推理与定位智能体**：综合模型结果、代码特征和知识库证据，对漏洞类型、根因和位置进行分析；
- **验证智能体**：通过规则、测试、不变量或攻击路径验证漏洞是否真实存在；
- **修复智能体**：生成适合当前上下文的修复方案；
- **报告智能体**：输出统一、可解释的安全分析报告。

当前已经存在一个基于 **Deep Lake** 的向量知识库，主要内容是不良实践。后续目标是最大化复用现有不良实践知识，并将新增知识全部进行向量化，形成一个统一、多视图、可复用的智能合约安全知识库。

---

## 2. 知识库的总体定位

现有知识库应从单纯的“不良实践库”扩展为：

> **智能合约安全反模式、漏洞判定、定位、验证、修复与报告的一体化向量知识库。**

知识库中的内容不应只说明“某种写法不好”，而应形成如下完整知识链：

```text
不良实践
    ↓
可能关联的漏洞
    ↓
漏洞成立条件
    ↓
排除条件或误报条件
    ↓
代码级证据与漏洞位置
    ↓
验证方法
    ↓
修复策略
    ↓
修复后回归验证
    ↓
最终报告表达
```

其中，向量主要用于相似度检索；原始文本、代码和少量元数据仍应保留，用于结果回传、证据追踪和知识管理。

## 3. 需要补充的知识类型

### 3.1 漏洞定义与判定知识

该类知识主要服务于 ChainSafeFort、推理定位智能体和验证智能体。

每种漏洞应包括：

- 漏洞名称与统一内部编号；
- 漏洞别名；
- 漏洞原理；
- 必要条件；
- 辅助证据；
- 攻击者所需能力；
- 可能造成的影响；
- 排除条件；
- 常见误报情形；
- 适用的 Solidity 或 EVM 版本；
- 相关不良实践；
- 推荐验证方法；
- 推荐修复方式。

示例：

```text
漏洞类型：重入漏洞

必要条件：
1. 函数发起外部调用；
2. 外部调用目标可执行攻击者代码；
3. 关键状态在外部调用前尚未更新；
4. 回调过程中可以重新进入敏感函数；
5. 重复执行能够造成资产损失或状态破坏。

排除条件：
1. 存在有效的重入锁；
2. 所有关键状态已在外部调用前更新；
3. 外部目标不可执行回调；
4. 重复调用不会造成安全影响。
```

---

### 3.2 漏洞代码案例

代码案例应至少包括三类：

#### 真实漏洞正样本

包括：

- 完整函数或关键代码片段；
- 漏洞所在语句；
- 相关状态变量；
- 外部调用；
- 攻击路径；
- 漏洞根因；
- 影响范围；
- 对应审计结论。

#### 修复后安全样本

包括：

- 修复后的函数代码；
- 修改位置；
- 修复原理；
- 修复是否影响接口、业务逻辑或 Gas；
- 修复后验证结果。

#### 困难负样本

困难负样本是与漏洞代码表面相似、但实际上安全的代码，例如：

- 存在 `call`，但状态已提前更新；
- 存在 `delegatecall`，但目标地址不可控；
- 存在敏感操作，但受到有效权限校验；
- 代码模式危险，但分支不可达；
- Solidity 版本已经提供默认安全检查；
- 存在重入保护或业务约束。

安全和困难负样本对降低推理智能体及验证智能体的误报非常重要。

---

### 3.3 函数级定位知识

该类知识用于将 Function Scoring Agent 的高风险函数进一步定位到语句级或变量级。

建议包括：

- 危险源语句；
- 状态变量读取位置；
- 状态变量写入位置；
- 外部调用位置；
- 权限检查位置；
- 返回值检查位置；
- 修饰器；
- 关键变量定义；
- 跨函数依赖；
- 调用前后执行顺序；
- 建议的定位范围。

示例：

```text
source：msg.sender.call{value: amount}("")
state_read：balances[msg.sender]
state_write：balances[msg.sender] -= amount
dangerous_order：external_call 发生在 state_write 之前
location：withdraw 函数第 8 至 12 行
```

---

### 3.4 跨合约与调用图模式知识

该类知识主要服务于 GCN 和推理定位智能体。

建议覆盖：

- 外部调用关系；
- 回调关系；
- 重入边；
- `delegatecall`；
- 代理合约与实现合约；
- 继承关系；
- 接口调用；
- 跨合约状态依赖；
- 跨合约权限传递；
- 资金流转；
- 返回值传播；
- 跨函数调用链；
- 跨合约攻击路径。

图结构可转换成规范化自然语言后向量化，例如：

```text
ContractA.withdraw 通过 external_call 调用 ContractB.receive，
ContractB.receive 通过 callback 重新进入 ContractA.withdraw，
ContractA 的用户余额在回调完成后才更新。
```

同时可保留少量元数据：

```json
{
  "scope": "cross_contract",
  "edge_types": ["external_call", "callback", "reentry"],
  "vulnerable_node": "ContractA.withdraw"
}
```

---

### 3.5 面向 DeepSVNN 的未知风险知识

DeepSVNN 只能检测异常，无法直接确定漏洞类型。因此知识库应重点补充以下两类知识。

#### 安全不变量

例如：

- 合约资产应覆盖所有用户余额；
- 只有管理员可以修改关键权限变量；
- 初始化函数只能执行一次；
- 提款后用户余额必须同步减少；
- 同一订单不能重复执行；
- 暂停状态下不能转移资产；
- 状态机只能按照合法路径迁移；
- 升级操作必须满足权限与存储布局约束。

#### 异常状态转换模式

例如：

- 先转账，后扣减余额；
- 权限变量修改后未重新校验；
- 外部调用失败但状态仍提交；
- 状态机跳过必要中间状态；
- 同一凭证可重复使用；
- 多函数组合导致资产重复释放；
- 多次初始化；
- 代理升级后存储冲突。

DeepSVNN 检测异常后，可根据函数行为、状态读写、调用序列和资金流检索相似安全反模式，而不是直接检索具体漏洞名称。

---

### 3.6 验证知识

验证智能体应基于可执行或可检查的证据，而不是单纯让大语言模型再次阅读代码。

验证知识应包含：

- 验证目标；
- 漏洞成立前提；
- 测试环境；
- 测试输入；
- 攻击合约模板；
- 调用顺序；
- 预期异常结果；
- 预期安全结果；
- 漏洞成立判据；
- 漏洞不成立判据；
- 无法验证的原因；
- 修复后回归测试方法。

验证方式可以包括：

- 静态规则检查；
- 数据流分析；
- 调用图分析；
- Slither 检测；
- Foundry 单元测试；
- Foundry invariant testing；
- Echidna 模糊测试；
- 符号执行；
- 污点分析；
- 攻击合约复现；
- 修复前后差分测试。

验证状态可统一为：

```text
verified_positive：成功复现漏洞
verified_negative：验证后未发现漏洞
inconclusive：依赖或环境不足，无法得出结论
not_executed：尚未执行验证
```

---

### 3.7 修复知识

修复知识不应只包含固定模板，还应包含适用条件与副作用。

每条修复知识建议包括：

- 对应漏洞或不良实践；
- 适用条件；
- 推荐修改位置；
- 修复代码模板；
- 替代方案；
- 不适用情况；
- 对业务逻辑的影响；
- 对接口的影响；
- Gas 影响；
- Solidity 版本兼容性；
- 可能引入的新风险；
- 修复后的验证方法。

例如，重入风险可以对应多种修复策略：

- 检查—生效—交互模式；
- `nonReentrant` 重入锁；
- 提款模式；
- 限制外部调用目标；
- 拆分状态更新和资产转移；
- 调整跨合约交互流程。

---

### 3.8 模型输出语义知识

不同检测模型输出的含义不同，推理智能体必须能够正确解释这些结果。

#### ChainSafeFort

知识内容包括：

- 四个 LSTM 分别检测的漏洞类型；
- 输出分数的语义；
- 建议阈值；
- 常见误报和漏报；
- 是否支持定位；
- 适用输入范围。

#### GCN

知识内容包括：

- GCN 检测的是跨合约风险；
- 节点、边和粗定位结果的含义；
- 漏洞类型输出是明确分类还是候选类别；
- 图结构不完整时的局限。

#### DeepSVNN

知识内容包括：

- 异常分数的含义；
- 训练分布范围；
- 高异常分数不等于确认存在漏洞；
- 特殊业务逻辑可能造成异常；
- 无法直接输出漏洞类型。

#### Function Scoring Agent

知识内容包括：

- 函数评分由哪些特征构成；
- 高分只表示优先检查；
- 不同函数分数是否可直接比较；
- Top-K 的选择逻辑；
- 评分置信度。

---

### 3.9 报告与风险评级知识

报告智能体应使用统一标准输出结论。

建议区分以下状态：

```text
Detected：检测模型发现风险
Suspected：推理认为可能存在漏洞
Verified：验证证明漏洞成立
Rejected：验证后认为是误报
Unverified：由于依赖或环境限制尚无法确认
```

每条报告知识应包括：

- 漏洞名称或候选风险；
- 检测来源；
- 涉及合约、函数和代码位置；
- 触发条件；
- 证据链；
- 验证结果；
- 风险影响；
- 严重性；
- 修复建议；
- 置信度；
- 不确定性说明。

## 4. 知识来源

### 4.1 官方标准与语言文档

用于构建漏洞定义、判定条件、版本语义和安全规则。

推荐来源：

- OWASP Smart Contract Security；
- EEA EthTrust Security Levels；
- Solidity 官方安全文档；
- SWC Registry；
- EVM 与 Solidity 版本文档。

适合提取：

- 漏洞定义；
- 必要条件；
- 排除条件；
- 安全要求；
- 版本差异；
- 官方修复建议。

SWC Registry 可用于兼容旧标签，但不应作为唯一的最新知识来源。

---

### 4.2 专业安全机构与审计报告

用于构建真实漏洞、不良实践、修复方式和验证知识。

推荐来源：

- Trail of Bits；
- Building Secure Contracts；
- Not So Smart Contracts；
- OpenZeppelin Contracts；
- OpenZeppelin 安全审计报告；
- 其他公开审计机构报告。

适合提取：

- 漏洞描述；
- 攻击路径；
- 漏洞代码；
- 安全代码；
- 修复方式；
- 修复状态；
- 项目方响应；
- 严重性评级。

---

### 4.3 开源漏洞数据集

#### SmartBugs Curated

适合：

- 小规模人工整理漏洞；
- 建立初始函数级样本；
- 训练或评测四个 LSTM；
- 构建检索测试集。

#### DAppSCAN

适合：

- 真实 DApp 源码；
- 审计报告；
- 漏洞 Finding；
- 源码与漏洞位置对齐；
- 跨合约漏洞案例。

#### SolidiFI

适合：

- 合成漏洞；
- 扩充代码模式；
- 压力测试；
- 测试模型是否能识别局部特征。

合成数据应标记为合成样本，不能与专家审计确认的真实漏洞赋予相同证据权重。

---

### 4.4 安全工具文档

用于构建验证方法和可执行测试知识。

推荐来源：

- Slither；
- Echidna；
- Foundry；
- 其他符号执行、污点分析或测试工具。

适合提取：

- 静态检测规则；
- AST 和调用图特征；
- 状态变量读写；
- 安全不变量；
- 模糊测试方式；
- 攻击测试模板；
- 判定标准。

---

### 4.5 修复前后提交记录

公开项目的修复提交是高价值知识来源。

建议建立如下关联：

```text
审计 Finding
    ↓
项目版本或 commit
    ↓
漏洞代码
    ↓
修复 commit
    ↓
修复后代码
    ↓
修复说明
    ↓
回归验证
```

这类数据特别适合服务修复智能体和验证智能体。

## 5. 知识处理流水线

建议采用如下统一处理流程：

```text
知识采集
    ↓
保存原始证据
    ↓
内容解析与清洗
    ↓
按知识单元切分
    ↓
统一漏洞分类
    ↓
代码与报告对齐
    ↓
生成多视图知识
    ↓
质量校验
    ↓
向量化
    ↓
写入 Deep Lake
```

## 6. 保存原始证据

所有资料在处理前应保留原始文件。

推荐目录结构：

```text
raw/
├── standards/
├── solidity_docs/
├── audit_reports/
├── vulnerable_contracts/
├── safe_contracts/
├── test_cases/
└── datasets/
```

每个来源应记录：

```json
{
  "source_id": "SRC-0001",
  "source_name": "DAppSCAN",
  "source_type": "audit_dataset",
  "source_version": "commit 或版本",
  "retrieved_at": "采集日期",
  "license": "许可证",
  "content_hash": "SHA256",
  "original_path": "原始文件路径"
}
```

保留原始证据的目的包括：

- 可追溯；
- 可复查；
- 可重新切分；
- 可重新生成向量；
- 可处理版本变化；
- 可校验大模型提取结果；
- 可进行许可证审查。

---

## 7. 知识切分策略

### 7.1 标准和安全文档

按照完整规则切分，而不是固定字数切分。

一个知识单元应尽量包含：

```text
规则名称
漏洞定义
成立条件
排除条件
影响
修复建议
```

---

### 7.2 审计报告

按照一个 Finding 切分。

一个 Finding 应包括：

```text
漏洞标题
严重性
漏洞描述
代码位置
攻击路径
影响
修复建议
项目响应
修复状态
```

---

### 7.3 Solidity 源码

建议按照以下层级切分：

```text
项目
  → 合约
    → 函数
      → 关键代码片段
```

函数级知识应尽量保留：

- Solidity 版本；
- 状态变量定义；
- 函数修饰器；
- 父合约；
- 被调用函数；
- 外部调用；
- 必要的上下文；
- 关键依赖。

只保存一条危险语句通常不足以判断是否真正存在漏洞。

---

### 7.4 跨合约案例

跨合约案例应按照完整调用链或攻击路径切分：

```text
入口函数
  → 中间合约
  → 外部调用
  → 回调或 delegatecall
  → 状态变化
  → 风险结果
```

---

## 8. 统一漏洞分类

不同数据源可能使用不同名称，例如：

```text
Reentrancy
Re-entrancy
重入攻击
SWC-107
OWASP SC08
```

建议映射到统一内部编号：

```json
{
  "internal_id": "VULN_REENTRANCY",
  "canonical_name": "Reentrancy",
  "aliases": [
    "Re-entrancy",
    "重入攻击"
  ],
  "swc_id": "SWC-107",
  "owasp_id": "对应 OWASP 编号"
}
```

四个 LSTM 与漏洞类别的对应关系属于固定系统配置，不应通过向量检索动态决定。

示例：

```text
LSTM_1 → VULN_REENTRANCY
LSTM_2 → VULN_ACCESS_CONTROL
LSTM_3 → VULN_INTEGER_ARITHMETIC
LSTM_4 → VULN_UNCHECKED_CALL
```

具体映射应按照实际四种漏洞配置。

---

## 9. 最大化复用现有不良实践

现有 Deep Lake 中的每条不良实践应保留为父知识，不需要删除或替换。

例如：

```text
BP-001：
外部调用后才更新余额是一种不良实践。
```

可扩展为：

```text
BP-001
├── definition_view
├── behavior_view
├── necessary_condition_view
├── exclusion_condition_view
├── vulnerable_code_view
├── safe_code_view
├── localization_view
├── cross_contract_view
├── invariant_view
├── verification_view
├── repair_view
└── report_view
```

所有子知识通过同一个 `parent_id` 关联到原始不良实践。

## 10. 多视图知识生成

### 10.1 定义视图

```text
关键状态更新发生在外部调用之后，是一种高风险的交互顺序。
```

### 10.2 行为视图

```text
函数向用户可控地址发起外部调用，并在调用返回后修改用户余额。
```

### 10.3 漏洞成立条件视图

```text
当外部地址可执行回调、关键状态尚未更新，并且回调能够重复执行敏感操作时，该模式可能形成可利用的重入漏洞。
```

### 10.4 排除条件视图

```text
如果函数受到有效重入锁保护，或关键状态已经在外部调用前更新，则相似代码不一定构成重入漏洞。
```

### 10.5 定位视图

```text
检查外部调用、调用前余额读取和调用后余额写入之间的执行顺序。
```

### 10.6 验证视图

```text
构造带 fallback 或 receive 回调的攻击合约，在回调期间再次调用目标提款函数，检查是否可重复提款。
```

### 10.7 修复视图

```text
优先在外部调用前更新关键状态，并根据调用路径增加有效重入保护。
```

### 10.8 跨合约视图

```text
目标合约向外部合约转账，外部合约通过回调重新进入目标合约，而余额更新尚未完成。
```

## 11. 代码与审计报告对齐

真实审计知识应尽量建立以下关系：

```text

    ↓

    ↓
合约文件
    ↓
函数签名
    ↓
漏洞行范围
    ↓
```

推荐字段：

```json
{
  "finding_id": "FINDING-001",
  "repository_commit": "abc123",
  "contract_name": "Vault",
  "function_signature": "withdraw(uint256)",
  "start_line": 81,
  "end_line": 96,
  "vulnerable_code": "...",
  "patched_commit": "def456",
  "safe_code": "...",
  "verification_status": "expert_confirmed",
  "location_confidence": "high"
}
```

如果报告没有明确代码位置，应将位置置信度标记为低，不能让大模型自行生成不存在的行号。

## 12. 使用大语言模型进行知识提取

可使用大语言模型将报告或文档转换为结构化数据，例如：

```json
{
  "title": "",
  "knowledge_type": "",
  "vulnerability_type": "",
  "bad_practice": "",
  "necessary_conditions": [],
  "exclusion_conditions": [],
  "affected_functions": [],
  "dangerous_operations": [],
  "impact": "",
  "verification_method": "",
  "repair_method": ""
}
```

但大语言模型的输出必须经过确定性校验：

1. 漏洞类型必须存在于统一分类表；
2. 函数名必须在源码中真实存在；
3. 行号不得超过文件范围；
4. Solidity 版本必须来自源码或项目配置；
5. SWC、OWASP 等编号必须通过映射表验证；
6. 引用必须能回溯到原文；
7. 缺失信息保存为 `unknown`；
8. 真实漏洞和合成漏洞必须分开标记；
9. 不允许模型自行补造攻击结果或修复状态。

还应记录：

```text
extraction_model
prompt_version
extraction_time
review_status
```

## 13. 安全负样本生成

安全样本可来自：

- OpenZeppelin 安全实现；
- 审计报告中的修复后代码；
- 项目的 patched commit；
- 有危险操作但有有效防护的代码；
- 不满足漏洞成立条件的相似函数。

安全样本应说明“为什么安全”：

```json

  "sample_type": "hard_negative",
  "looks_risky_because": [
    "存在外部 call"
  ],
  "safe_because": [
    "关键余额已提前更新",
    "存在有效重入锁"
  ]
}
```

## 17. 不同模型的检索策略

### 17.1 ChainSafeFort

ChainSafeFort 已经给出候选漏洞类型，因此检索查询可包含：

```text
漏洞类型
+ 可疑函数代码摘要
+ 函数危险行为
+ LSTM 预测结果
```

优先检索：

- 漏洞定义；
- 成立条件；
- 误报条件；
- 相似代码；
- 验证方法；
- 修复方案。

---

### 17.2 GCN

GCN 检索查询可包含：

```text
可疑合约节点
+ 可疑函数
+ 调用边
+ 调用链摘要
+ 状态读写关系
+ 粗略漏洞类型
```

优先检索：

- 跨合约图模式；
- 调用链案例；
- 状态依赖；
- 跨合约攻击路径；
- 代理和回调模式。

---

### 17.3 DeepSVNN

DeepSVNN 不应以具体漏洞名称为核心查询。

查询应包含：

```text
异常函数行为
+ 状态变量读写
+ 外部调用
+ 权限变化
+ 资金流
+ 调用序列
+ 状态转换
```

优先检索：

- 不良实践；
- 安全不变量；
- 异常状态转换；
- 业务逻辑风险；
- 未知漏洞案例；
- 相似异常行为。

检索结果只能作为候选解释，不能直接将未知异常强行分类为某种已知漏洞。

## 19. 知识质量评分

每条知识应建立质量与证据等级。

建议考虑：

- 来源是否官方；
- 是否来自专家审计；
- 是否是真实项目；
- 是否有人审；
- 是否有准确代码定位；
- 是否有修复前后对照；
- 是否可执行验证；
- 是否适用于当前版本；
- 是否为合成样本；
- 是否为自动工具标签。

推荐等级：

| 等级 | 含义 |
|---|---|
| A | 官方标准或专家审计，并有源码与验证 |
| B | 人工标注的真实漏洞案例 |
| C | 学术整理、公开数据集或合成漏洞 |
| D | 工具自动判断或未经人工核验的大模型提取 |

推理和报告阶段应优先使用 A、B 级知识。

D 级知识只能作为线索，不能单独支持“漏洞已确认”的结论。

## 20. 不适合放入向量知识库的内容

以下内容属于系统编排或配置，不应依赖向量检索决定：

- 模型调用顺序；
- 四个 LSTM 与漏洞类型的固定映射；
- 各模型阈值；
- Top-K 数量；
- 智能体重试次数；
- 是否进入验证流程；
- Agent 输入输出 JSON 格式；
- 工作流状态机；
- 权限控制；
- 任务超时；
- 最终决策逻辑。

这些内容应保存在配置文件、Router 或多智能体编排程序中。

当前用户提交的合约、模型输出、函数评分和调用图属于本次任务的短期上下文，不属于静态知识，应与 Deep Lake 的检索结果一起传递给推理智能体。

## 23. 总结

现有 Deep Lake 不良实践库可以作为整个知识系统的基础，不需要重新建设多套独立知识库。

最合适的扩展方式是：

> 保留原有不良实践，将每条不良实践作为父知识，进一步扩展为漏洞定义、成立条件、排除条件、代码案例、安全对照、函数定位、跨合约模式、安全不变量、验证方法、修复方案和报告表达等多种向量视图。

知识获取应优先来自：

1. 官方标准与 Solidity 文档；
2. 专业安全机构和审计报告；
3. 真实漏洞数据集；
4. 修复前后项目提交；
5. 安全工具文档；
6. 高质量安全实现。

知识处理应遵循：

```text
原始证据保留
→ 按语义单元切分
→ 统一分类
→ 代码与报告对齐
→ 生成多视图知识
→ 质量校验
→ 全量向量化
→ 写入 Deep Lake
```

最终形成一个可同时服务 ChainSafeFort、GCN、DeepSVNN、定位、验证、修复和报告智能体的统一安全知识库。
