# 单用户智能合约安全检测系统构建方案

## 1. 系统定位

本系统定位为面向单一用户的一站式智能合约安全检测、定位、验证、修复与报告平台。

系统不再区分普通用户、审计员和管理员。用户可以在同一界面中查看并操作：

- 四个 LSTM 已知漏洞检测结果；
- GCN 跨合约检测结果；
- DeepSVNN 未知风险结果；
- 函数风险评分；
- 大语言模型推理与定位结果；
- Deep Lake 知识库检索依据；
- 漏洞验证结果；
- 修复建议与代码 Diff；
- 其他潜在风险警告；
- 最终检测报告。

系统通过“基础视图 / 高级视图”控制信息复杂度，而不是通过角色权限控制内容可见性。

```text

    ↓

    ↓

    ↓

    ↓
```

```text

Redis：任务队列、缓存、实时进度
MinIO：源码、日志、测试产物和报告
```

## 3. 用户检测模式

| 检测模式 | 调用路径 |
|---|---|
| 指定漏洞检测 | 对应 LSTM → 风险评分 → RAG → 推理定位 → 可选验证与修复 |
| 已知漏洞全检 | 四个 LSTM 并行 → 风险评分 → 推理定位 → 验证 |
| 跨合约检测 | GCN + 相关 LSTM → 调用图推理与定位 |
| 未知风险检测 | DeepSVNN + 四个 LSTM 排除已知类型 → 推理 |
| 完整安全分析 | 四个 LSTM + GCN + DeepSVNN → 推理、验证、修复 |

所有专项检测默认执行其他潜在风险筛查：

```text
全局函数风险评分
→ Deep Lake 通用检索
→ 推理模型筛查其他风险
→ 只生成范围外警告
→ 提示用户重新发起对应专项检测
```

范围外风险不直接进入验证和修复流程。

## 15. 工作流路由器

路由器使用确定性规则，根据用户选择决定模型路径。

```python
def route_audit(request, project):
    plan = []

    if request.mode == "specified_vulnerability":
        plan.append(get_target_lstm(request.target_vulnerability))
    elif request.mode == "known_full_scan":
        plan.extend(get_all_lstm_models())
    elif request.mode == "unknown_risk_scan":
        plan.extend([*get_all_lstm_models(), "DEEPSVNN"])
    elif request.mode == "full_audit":
        plan.extend([*get_all_lstm_models(), "DEEPSVNN"])

    if project.is_multi_contract and request.include_cross_contract:
        plan.append("GCN")

    return plan
```

专项检测始终追加：

```text
Function Risk → Deep Lake → Reasoning Agent → Warning Gate
```

## 16. 模型 Adapter 与统一证据

统一接口：

```python
class DetectionAdapter:
    async def analyze(self, context) -> list:
        raise NotImplementedError
```

实现四个 LSTM Adapter、GCNAdapter 和 DeepSVNNAdapter。

统一输出：

```json
{
  "model_id": "LSTM_REENTRANCY",
  "scope": "function",
  "contract": "Vault.sol",
  "function": "withdraw(uint256)",
  "vulnerability_id": "VULN_REENTRANCY",
  "raw_score": 0.91,
  "normalized_score": 0.84,
  "feature_evidence": []
}
```

DeepSVNN 的 `vulnerability_id` 为空，只表示异常风险。

统一证据中心负责：分数归一化、结果去重、函数级证据合并、证据链保存以及为风险函数和推理模型提供输入。

## 17. 风险评分模块

实现三个分数：


## 19. 大语言模型智能体

### 推理与定位智能体

读取源码、调用图、模型证据、风险分数、Solidity 版本和知识库结果，输出：

```json
{
  "in_scope_findings": [],
  "out_of_scope_warnings": []
}
```

限制：不得直接标记为已验证；范围外风险只生成警告；必须引用证据。

### 验证智能体

```text
候选漏洞
→ 生成验证计划
→ 调用 Slither / Foundry / Echidna
→ 执行测试
→ 解析日志
→ 输出 verified_positive / verified_negative / inconclusive
```

### 修复智能体

```text
已验证漏洞
→ 检索修复知识
→ 生成补丁
→ 编译
→ 漏洞回归测试
→ 项目回归测试
→ 返回代码 Diff
```

> 单用户界面 + 动态模型路由 + 多模型证据融合 + 函数风险排序 + Deep Lake RAG + 推理定位 + 工具验证 + 智能修复 + 模板报告。

## 27. 后端现状对照与补充清单（2026-07-03）

> 本节用于区分“当前后端已经具备的能力”和“为了支撑单用户 Web 工作台还需要补充的能力”。当前后端已经可以完成上传源码、多模型检测、风险评分、报告生成、任务管理、Supabase/PostgreSQL 持久化和基础用户认证；后续重点应从“检测主链路”转向“前端工作台需要的细粒度数据接口、实时状态、验证与修复闭环”。


### 27.2 第一优先级：前端工作台直接依赖的接口

当前主要通过 `/api/v1/audits/{taskId}/report` 返回完整报告。为了让前端页面按模块展示，需要补充细粒度只读接口：

```http
GET /api/v1/audits/{taskId}/summary
GET /api/v1/audits/{taskId}/workflow
GET /api/v1/audits/{taskId}/functions/ranking
GET /api/v1/audits/{taskId}/findings
GET /api/v1/audits/{taskId}/findings/{findingId}
GET /api/v1/audits/{taskId}/warnings
GET /api/v1/audits/{taskId}/call-graph
```

这些接口可以先从现有完整报告 JSON 中拆分生成，不必一开始就重构数据库结构。这样前端可以较快完成仪表盘、漏洞详情、风险排名、模型证据和警告列表展示。

### 27.3 第一优先级：合约文件树与代码定位接口

方案中的 Monaco Editor 需要后端提供文件、函数、漏洞位置和标记信息。当前后端已经能读取和预处理 Solidity 源码，但还没有对外暴露文件树和行级标记接口。

需要补充：

```http
GET /api/v1/audits/{taskId}/contracts/tree
GET /api/v1/audits/{taskId}/files/{fileId}
GET /api/v1/audits/{taskId}/files/{fileId}/markers
```

建议返回内容包括：源码文件树、合约列表、函数列表、函数起止行、漏洞或警告所在行、证据跳转位置、修复前后 Diff 对应位置。

### 27.4 第一优先级：更细的任务阶段状态

当前任务状态偏粗，主要是 `created / queued / running / cancelling / succeeded / failed / cancelled / interrupted`。

为了支撑前端“模型执行流程”展示，建议补充 workflow stage：

```text
preprocessing
lstm_detecting
gcn_detecting
deepsvdd_detecting
evidence_merging
scoring
rag_retrieval
reasoning
verifying
repairing
reporting
completed
failed
skipped
```

注意区分：

```text
task.status：任务总体状态，用于任务列表。
workflow_stages：阶段级状态，用于前端流程图和进度条。
events：状态变化日志，用于追踪和调试。
```

### 27.5 第一优先级：WebSocket 实时进度推送

当前前端需要通过轮询任务状态获取进度。方案中要求：

```text
ws://server/api/v1/ws/audits/{taskId}
```

建议补充 WebSocket，用于推送：

```json
{
  "task_id": "TASK-xxx",
  "stage": "scoring",
  "status": "running",
  "progress": 55,
  "message": "Computing function risk vectors",
  "time": "2026-07-03T..."
}
```

短期可以基于内存事件广播实现；中长期如果接 Redis/Celery，再改为 Redis Pub/Sub 或任务队列事件。

### 27.6 第一优先级：数据库细粒度拆表

当前 Supabase/PostgreSQL 已经能存用户、任务、事件、报告和附件，但大量检测结果仍主要保存在完整 `report_json` 中。前端初期可以使用，但后续做筛选、统计、历史查询和详情页会不方便。

建议逐步补充以下表：

```text
projects
project_files
functions
workflow_stages
model_runs
model_evidence
function_risks
findings
warnings
knowledge_references
```

拆表优先级建议：

```text
functions
model_evidence
function_risks
findings
warnings
workflow_stages
knowledge_references
```

这样前端可以直接查询函数风险排名、证据详情、Finding/Warning 详情，而不必每次解析完整报告 JSON。

### 27.7 第二优先级：项目管理

当前后端以“审计任务”为中心，方案中则有“项目 - 文件 - 审计任务”的层级。

需要补充：

```http
POST /api/v1/projects
GET  /api/v1/projects
GET  /api/v1/projects/{projectId}
POST /api/v1/projects/{projectId}/upload
GET  /api/v1/projects/{projectId}/audits
```

项目层可以保存项目名称、项目描述、上传文件列表、最近一次审计任务、历史审计任务和项目级风险统计。

如果比赛时间紧，可以先不做完整项目管理，只保留任务历史；但如果前端有 `/projects` 页面，后端需要补这部分。

### 27.8 第二优先级：大模型真实调用

当前 `.env` 已经支持：

```env
SCG_LLM_PROVIDER=
SCG_LLM_BASE_URL=
SCG_LLM_API_KEY=
SCG_LLM_MODEL=
SCG_LLM_TEMPERATURE=
```

但 `backend/agents/llm_client.py` 中 `OpenAICompatibleLLMClient.complete_json()` 仍是预留接口。需要补充真实 HTTP 调用逻辑，并要求大模型严格返回 JSON。

需要保证：大模型不能修改上游模型分数；不能把未验证结果标记为已验证；必须引用证据；范围外风险只能进入 warning。

### 27.9 第二优先级：验证智能体

方案中验证智能体需要接入 Slither / Foundry / Echidna。当前后端还没有真正执行这些工具。

需要补充：验证任务创建、验证计划生成、工具执行、日志解析、验证结果归一化和验证记录持久化。

验证结果状态建议固定为：

```text
verified_positive
verified_negative
inconclusive
tool_failed
skipped
```

同时必须保证：未经过工具或人工验证的漏洞不能显示为“已验证”。

### 27.10 第二优先级：修复智能体与代码 Diff

当前后端可以给出修复建议，但还没有形成完整修复闭环。

需要补充：

```http
POST /api/v1/audits/{taskId}/findings/{findingId}/repair
GET  /api/v1/audits/{taskId}/repairs/{repairId}
```

修复智能体应输出补丁说明、unified diff、修改后的源码、编译结果、漏洞回归测试结果、项目回归测试结果和修复状态。

修复状态建议：

```text
suggested
patch_generated
compile_passed
compile_failed
regression_passed
regression_failed
applied
rejected
```

必须保证：未通过编译和回归测试的补丁不能显示为“修复成功”。

### 27.11 第二优先级：Deep Lake 向量知识库

当前知识库更接近 JSONL 检索，可以支撑早期 RAG。方案中明确写 Deep Lake，因此后续如果要贴合方案，需要把知识库升级为真正向量库。

建议保留统一接口：

```python
async def rag_search(
    query_text: str,
    view_types: list[str],
    vulnerability_ids: list[str] | None,
    top_k: int = 10,
):
    ...
```

这样底层可以先 JSONL，后面替换为 Deep Lake，不影响推理智能体。

### 27.12 第三优先级：报告导出增强

当前已有 JSON 和 Markdown 报告。方案中还需要：

```http
POST /api/v1/audits/{taskId}/report/export
```

导出格式建议：`json / markdown / html / pdf`。PDF 可以放到后期做，早期先补 HTML 会更容易服务前端预览。

### 27.13 第三优先级：异步队列与部署基础设施

当前异步执行使用 FastAPI 进程内 `ThreadPoolExecutor`，开发阶段够用，但部署稳定性一般。

后续可升级为：

```text
Redis + Celery / RQ
```

同时补充 Docker Compose、worker 服务、Redis、MinIO、模型服务拆分和验证工具容器。

对象存储方面，当前源码、报告和日志仍保存在本地 `backend_outputs/`。如果要多人协作和部署，建议后续接 MinIO 或 Supabase Storage。

### 27.14 推荐补充顺序

```text
1. 拆分报告查询接口：summary / findings / warnings / ranking / workflow / call-graph
2. 合约文件树、源码读取、代码 markers

4. workflow_stages 与 WebSocket 实时进度
5. 项目管理 projects
6. 大模型 OpenAI-compatible 真实调用

```
