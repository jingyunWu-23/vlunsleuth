# 多模型智能合约漏洞检测系统设计方案

## 1. 系统目标

系统集成以下检测模型：

- **四个 LSTM 二分类模型**：分别检测四类已知漏洞；
- **GCN 跨合约检测模型**：检测跨合约已知漏洞，并提供粗粒度位置或调用关系证据；
- **DeepSVNN**：检测未知漏洞或异常行为，但不直接输出漏洞类型；
- **大语言模型智能体**：负责推理、定位、验证方案生成和修复建议；
- **Deep Lake 向量知识库**：提供不良实践、漏洞条件、代码案例、验证方法和修复知识。

系统需要根据用户选择动态调度模型，同时保证：即使用户只选择某一类已知漏洞，也会对其他潜在风险进行后台筛查，但只给出警告，不直接验证。

---

## 2. 用户检测模式与模型路径

| 模式 | 主要调用路径 |
|---|---|
| 指定漏洞检测 | 对应 LSTM → 推理定位 → 可选验证与修复 |
| 已知漏洞全检 | 四个 LSTM 并行 → 推理定位 → 验证 |
| 跨合约检测 | GCN + 相关 LSTM → 调用链推理与定位 |
| 未知风险检测 | DeepSVNN + 四个 LSTM 排除已知类型 → 推理 |
| 完整安全审计 | 四个 LSTM + GCN + DeepSVNN → 推理、验证、修复 |

所有专项检测默认附加：

```text
全局函数风险筛查
→ Deep Lake 检索
→ 推理模型识别其他潜在风险
→ 仅生成范围外警告
→ 提示用户重新发起对应专项检测
```

## 4. 工作流路由器设计

路由器应采用确定性规则，不应完全依赖大语言模型。

```python
def build_workflow(request, project):
    workflow = {
        "formal_models": [],
        "background_screening": True,
        "agents": []
    }

    for vulnerability in request.target_vulnerabilities:
        workflow["formal_models"].append(
            get_lstm_model(vulnerability)
        )

    if request.mode == "known_full_scan":
        workflow["formal_models"] = ["LSTM_ALL"]

    if request.mode == "unknown_risk_scan":
        workflow["formal_models"] += ["DEEPSVNN", "LSTM_ALL"]

    if request.mode == "full_audit":
        workflow["formal_models"] += ["LSTM_ALL", "DEEPSVNN"]

    if project.is_multi_contract and request.mode in {
        "cross_contract_scan", "full_audit"
    }:
        workflow["formal_models"].append("GCN")

    workflow["agents"].append("reasoning_localization")

    if request.need_verification:
        workflow["agents"].append("verification")

    if request.need_repair:
        workflow["agents"].append("repair")

    workflow["agents"].append("report_service")
    return workflow
```

## 6. 大语言模型角色设计

第一阶段可采用：

> 同一个基础模型 + 不同系统提示词 + 不同工具权限 + 不同输出结构。

### 6.1 推理与定位智能体

输入：模型证据、Top-K 高风险函数、源码、调用图、Deep Lake 检索结果和 Solidity 版本。

职责：

- 综合不同模型证据；
- 判断候选漏洞类型；
- 分析漏洞成立条件；
- 排除明显误报；
- 定位危险语句；
- 对未知异常给出候选解释。

它只能输出 `suspected`，不能直接输出 `verified`。

### 6.2 验证智能体

验证流程：

```text
候选漏洞
→ 生成验证计划
→ 调用 Slither / Foundry / Echidna 等工具
→ 执行测试
→ 解析结果
→ 输出 Verified / Rejected / Inconclusive
```

### 6.3 修复智能体

输入：原始代码、漏洞根因、验证结果、RAG 修复知识和 Solidity 版本。

输出：修复策略、修改代码、修改位置、兼容性影响、可能副作用和回归测试方案。

### 6.4 报告生成

报告生成没有必要单独部署一个大模型。

```text
结构化 Finding
+ 验证结果
+ 修复结果
+ 检测范围
        ↓
固定模板
        ↓
Markdown / HTML / PDF
```

大模型只可选用于生成摘要，不能修改上游结论。

## 7. 指定漏洞检测与其他风险预警

当用户指定某类漏洞时，系统采用双轨流程。

```text
用户选择指定漏洞
          ↓
 ┌────────┴─────────┐
 │                  │
正式检测轨道       全局预警轨道
 │                  │
对应 LSTM          通用函数风险评分
 │                  │
专项 RAG           通用不良实践 RAG
 │                  │
推理定位           推理模型筛查其他风险
 │                  │
可选验证与修复     只生成警告
 └────────┬─────────┘
          ↓
        合并报告
```

### 正式检测结果

```json
{
  "scope": "in_scope",
  "status": "suspected",
  "verification_allowed": true,
  "repair_allowed": true
}
```

### 范围外风险警告

```json
{
  "scope": "out_of_scope",
  "status": "screening_warning",
  "verification_allowed": false,
  "repair_allowed": false,
  "recommended_action": {
    "action": "start_new_scan",
    "target_vulnerability": "VULN_ACCESS_CONTROL"
  }
}
```

报告应明确区分：

1. 本次正式检测结果；
2. 其他潜在风险提示；
3. 未覆盖范围与检测局限。

## 8. 风险函数总体设计

风险评分应分为三类：

```text
R_func
→ 通用函数风险排序

R_selected
→ 用户指定漏洞的正式评分

R_warning
→ 用户未选择漏洞的预警评分
```

---

## 9. 通用函数风险评分

对函数 \(f\)：

\[
R_{func}(f)=
clip(
w_aA_f+
w_gG_f+
w_sS_f+
w_bB_f+
w_kK_f+
w_cC_f-
w_pP_f,\ 0,\ 1)
\]

| 分量 | 含义 |
|---|---|
| \(A_f\) | DeepSVNN 异常分数 |
| \(G_f\) | GCN 跨合约结构风险 |
| \(S_f\) | AST、CFG、数据流静态危险特征 |
| \(B_f\) | 业务敏感度 |
| \(K_f\) | Deep Lake 不良实践与案例相似度 |
| \(C_f\) | 多证据一致性 |
| \(P_f\) | 安全防护证据 |

推荐初始权重：

\[
R_{func}=
0.25A_f+
0.15G_f+
0.20S_f+
0.15B_f+
0.15K_f+
0.10C_f-
0.20P_f
\]

该分数只表示函数是否值得优先检查，不表示漏洞存在概率。

---

## 10. 各分量设计

### 10.1 DeepSVNN 异常分数

优先使用函数级异常分数。若只能输出合约级异常分数，应结合函数重要性分配，但更推荐改造为函数级输出。

### 10.2 GCN 跨合约风险

\[
G_f =
\lambda_1N_f+
\lambda_2\max E_e+
\lambda_3mean(E_e)
\]

单合约输入时可令 \(G_f=0\)。

### 10.3 静态代码风险

可检测：

- `call`；
- `delegatecall`；
- 外部调用后更新状态；
- 未检查返回值；
- `tx.origin`；
- 资产转移；
- 权限修改；
- 升级操作；
- 循环中外部调用。

可采用饱和计算：

\[
S_f=1-\prod_i(1-s_i)
\]

### 10.4 业务敏感度

```text
只读函数                   0.05
普通状态修改               0.20
代币转移或提款             0.60
权限修改                   0.70
升级实现                   0.80
销毁合约或转移全部资产     1.00
```

### 10.5 知识库相似度

同时检索真实漏洞案例、不良实践、安全代码和困难负样本。

\[
K_f=
clip(
0.6K_f^+
+0.3K_f^{bp}
-0.4K_f^-,\ 0,\ 1)
\]

### 10.6 多证据一致性

仅统计独立证据，例如 DeepSVNN、GCN、静态规则、知识库案例和用户指定 LSTM。

### 10.7 安全防护分数

可识别：有效 `nonReentrant`、检查—生效—交互顺序、有效访问控制、返回值检查、固定可信目标、输入约束和暂停机制。

## 11. 用户指定漏洞的专项评分

对用户选择的漏洞 \(v\)：

\[
R_{selected}(f,v)=
0.50L_{f,v}
+0.15S_{f,v}
+0.10G_{f,v}
+0.10K_{f,v}
+0.10C_{f,v}
+0.05A_f
-0.20P_{f,v}
\]

该分数用于函数排序、推理定位和是否进入验证流程。

---

## 12. 范围外风险预警评分

对用户未选择的候选漏洞 \(c\)：

\[
R_{warning}(f,c)=
R_{func}(f)
\times
E_{category}(f,c)
\times
Q_{evidence}(f,c)
\]

类别证据：

\[
E_{category}=
0.35K_{category}
+0.30S_{category}
+0.20G_{category}
+0.15C_{category}
\]

证据质量示例：

```text
只有大模型推测                     0.2
大模型 + 不良实践匹配              0.5
静态特征 + 知识库案例              0.7
模型证据 + 静态特征 + 知识库        0.9
```

建议预警触发条件：

```text
R_func >= 0.60
E_category >= 0.65
至少有两类独立证据
候选类型不属于用户当前选择
不存在明显安全防护或反例
```

分级建议：

| 预警分数 | 处理 |
|---|---|
| `< 0.40` | 不展示 |
| `0.40～0.60` | 仅记录日志 |
| `0.60～0.75` | 中低风险提示 |
| `>= 0.75` | 明确建议重新专项检测 |

---

## 13. 两阶段筛选流程

### 第一阶段：快速排序

```text
所有函数
→ 计算 R_func
→ 选择 Top-K
```

建议：

```text
小项目：Top 10
中型项目：Top 20
大型项目：前 10%，最多 50 个函数
```

### 第二阶段：候选分类

```text
Top-K 函数
→ Deep Lake 检索
→ 推理模型生成候选风险
→ 程序计算 R_warning
→ 决定是否展示警告
```

大语言模型负责提出候选类别、整理证据和解释原因；程序负责分数计算、阈值判断和流程控制。

```text
DETECTED
→ REASONED
→ SUSPECTED
→ VERIFIED / REJECTED / INCONCLUSIVE
→ PATCH_GENERATED
→ PATCH_VERIFIED
→ REPORTED
```

范围外风险：

```text

→ 建议用户重新选择专项检测
```

范围外风险不能直接进入验证和修复流程。

## 16. 实施优先级

1. 定义统一的任务、证据、Finding 和 Warning 数据结构；
2. 为四个 LSTM、GCN、DeepSVNN 实现 Adapter；
3. 实现确定性 Workflow Router；
4. 建立模型分数归一化和证据中心；
5. 实现通用函数风险评分；
6. 接入 Deep Lake 多视图检索；
7. 接入推理定位智能体；
8. 接入验证工具与验证智能体；
9. 接入修复智能体；
10. 使用模板实现报告生成。

---

## 17. 核心结论

系统应采用：

> **规则路由 + 多模型检测 + 统一证据 + 三层风险评分 + RAG 推理 + 工具验证 + 模板报告**

其中：

- 检测模型负责发现风险；
- 风险函数负责排序和触发流程；
- 大语言模型负责解释、定位和生成验证方案；
- 验证工具负责确认漏洞；
- 修复智能体负责生成并测试补丁；
- 报告服务负责组织已有结论；
- 用户未选择的其他风险只生成警告，不直接验证。

## 18. 项目落地方案与实施顺序

结合当前项目结构，最终系统不应重写已有模型代码，而应在现有模型与数据资产外部增加一层统一的 Web 审计系统。系统以 Solidity 源码作为输入，以 `LG-DeepSVDD/DeepSVDD` 作为未知漏洞与异常检测主线，以 `.h5` LSTM 模型作为已知漏洞推理入口，以 `cross-contract_detection` 作为跨合约图检测入口，以 `dataset/knowledge/*.jsonl` 作为完整知识库内容来源，并用向量检索增强大模型召回能力。

### 18.1 最终系统边界

进入最终 Web 系统的模块：

```text
classification_vulnerabilities / LSTM .h5
→ 已知漏洞检测模型来源

LG-DeepSVDD/DeepSVDD
→ 未知漏洞与异常检测主线

cross-contract_detection
→ 跨合约调用关系与跨合约风险检测

dataset/knowledge/*.jsonl
→ RAG 完整知识库与大模型上下文来源

dataset/database 或后续向量库
→ Top-K 相似案例召回增强
```

不进入最终在线审计主流程的模块：

```text
unknown_detect
→ 历史实验模块，不作为最终未知检测入口

generated_smart_contracts
→ 训练增强模块，只在训练阶段使用，不在 Web 审计时加载

classification_vulnerabilities/*.py 训练脚本
→ 保留为训练与实验代码，正式系统只加载训练好的 .h5 模型推理
```

### 18.3 总体运行流程

最终 Web 系统的主流程为：

```text

        ↓

        ↓
Workflow Router 根据检测模式选择模型
        ↓

        ↓

        ↓

        ↓

        ↓

        ↓

        ↓
Web 页面展示风险总览、漏洞详情、证据链和修复建议
```

### 18.4 第一阶段：后端地基

第一阶段目标是让系统可以接收一次审计请求，并输出结构化空结果或模拟结果。此阶段不急于接入真实模型。

优先实现：

```text
backend/schemas
backend/router/workflow_router.py
backend/evidence/evidence_center.py
backend/function_risk/risk_score.py
backend/reporting/markdown_report.py
```

核心数据结构包括：

- `AuditRequest`：用户选择的检测模式、目标漏洞、是否需要验证、是否需要修复、上传源码路径；
- `AnalysisInput`：预处理后的合约、函数、源码片段、调用图、特征路径；
- `ModelEvidence`：LSTM、DeepSVDD、GCN 的统一证据格式；
- `Finding`：正式检测范围内的疑似或已验证漏洞；
- `Warning`：用户未选择漏洞类型的范围外风险预警；
- `AuditReport`：Web 展示与报告导出的最终结构。

这一阶段完成后，应能跑通：

```text
创建任务
→ 构建 workflow
→ 生成模拟 evidence
→ 计算风险分数
→ 输出 Markdown/JSON 报告
```

### 18.5 第二阶段：源码预处理

由于最终输入是源码，必须建立统一预处理层。该层负责把 `.sol` 文件转换成后续模型可使用的结构化输入。

优先实现：

```text
source_loader.py
→ 读取单文件、目录或压缩包中的 Solidity 源码

solidity_parser.py
→ 提取 contract、function、line range、源码片段、简单调用关系

feature_extractor.py
→ 生成 LSTM/DeepSVDD 所需的 opcode 或 embedding 输入
```

如果短期内源码到 opcode/embedding 的完整链路不稳定，可以先采用两级策略：

```text
可直接生成特征
→ 进入真实模型推理

暂时无法生成特征
→ 使用静态规则、知识库和跨合约解析先生成可展示结果
```

这样可以保证 Web 系统先能演示完整审计流程，再逐步补齐模型输入。

### 18.6 第三阶段：模型 Adapter 接入

所有模型都通过 Adapter 接入，不允许模型脚本直接控制主流程。

#### LSTMAdapter

职责：

- 加载四类已知漏洞的 `.h5` 模型；
- 接收预处理后的函数级特征；
- 输出 `L_{f,v}` 与统一 `ModelEvidence`；
- 支持指定漏洞检测和已知漏洞全检。

建议支持的漏洞类型：

```text
reentrancy
timestamp
delegatecall
SBunchecked_low_level_calls
```

#### DeepSVDDAdapter

职责：

- 以 `LG-DeepSVDD/DeepSVDD` 为唯一未知异常检测入口；
- 输出函数级或合约级异常分数 `A_f`；
- 不直接判断漏洞类型；
- 只输出 `unknown_anomaly` 类证据。

如果短期只能得到合约级异常分数，则先结合函数业务敏感度与静态风险将分数分配到函数级，但后续应优先改造成函数级输出。

#### GCNAdapter

职责：

- 调用 `cross-contract_detection` 中稳定的跨合约检测入口；
- 输出跨合约调用链、风险节点、风险边和 `G_f`；
- 在单合约检测时允许返回空结果，并令 `G_f=0`。

### 18.7 第四阶段：证据中心与风险函数

证据中心负责把不同模型与知识库结果合并到同一个函数视角下。

推荐聚合粒度：

```text
project
  └── contract
      └── function
          ├── lstm_evidence
          ├── deepsvdd_evidence
          ├── gcn_evidence
          ├── static_evidence
          ├── rag_evidence
          ├── risk_score
          └── findings / warnings
```

风险函数按三类实现：

```text
R_func
→ 对所有函数做通用高风险排序

R_selected
→ 对用户指定漏洞做正式检测评分

R_warning
→ 对用户未选择漏洞做范围外预警评分
```

这一阶段应保证：

- 正式检测结果可以进入推理、验证和修复；
- 范围外风险只能生成预警；
- 范围外风险不能直接进入验证和修复流程。

### 18.8 第五阶段：知识库与大模型智能体

知识库采用 JSONL 完整内容 + 向量召回增强的方式。

推荐流程：

```text
向量检索
→ 找到 Top-K knowledge_id
→ 回到 JSONL 读取完整记录
→ 按 agent_targets 和 knowledge_type 过滤
→ 组装给大模型智能体
```

JSONL 的作用：

- 保存完整漏洞案例；
- 保存漏洞原因、定位、验证、修复、报告字段；
- 作为大模型理解上下文的主数据源。

向量库的作用：

- 根据函数源码、危险 API、漏洞类型、语义标签召回相似案例；
- 只负责检索，不替代 JSONL 结构化内容。

智能体按阶段接入：

```text
reasoning_localization_agent
→ 解释模型证据、判断疑似漏洞、定位危险语句

verification_agent
→ 生成 Slither / Foundry / Echidna 验证计划

repair_agent
→ 生成修复策略、补丁建议和回归测试建议
```

第一版可以只实现推理定位和修复建议，验证工具可先输出验证计划，后续再接真实工具执行。

### 18.9 第六阶段：Web 页面

Web 第一版应以审计工作台为主，不做营销式首页。

建议页面模块：

```text
上传审计
→ 上传 .sol 文件或项目压缩包，选择检测模式、目标漏洞、是否需要修复建议

任务进度
→ 展示预处理、LSTM、DeepSVDD、GCN、RAG、智能体分析、报告生成状态

风险总览
→ 展示函数风险排行、漏洞类型分布、未知异常分数、跨合约风险、范围外预警

漏洞详情
→ 展示源码位置、模型证据、知识库相似案例、推理解释、验证计划、修复建议

报告导出
→ 导出 Markdown / HTML / PDF
```

前端只消费后端结构化 JSON，不直接理解模型输出。所有模型解释、风险评分和报告内容都由后端生成。

### 18.10 推荐实施顺序

建议按以下顺序推进：

1. 新建 `backend` 基础目录和统一数据结构；
2. 实现 `workflow_router.py`，支持五种检测模式；
3. 实现 `evidence_center.py`，统一存放模型证据；
4. 实现 `risk_score.py`，先使用可配置权重计算 `R_func`、`R_selected`、`R_warning`；
5. 实现 `jsonl_knowledge_store.py`，读取 `dataset/knowledge/unified_multi_agent_knowledge.jsonl`；
6. 实现 `source_loader.py` 和基础 Solidity 函数切分；
7. 实现 `LSTMAdapter`，加载 `.h5` 模型完成已知漏洞推理；
8. 实现 `DeepSVDDAdapter`，接入 `LG-DeepSVDD/DeepSVDD` 的异常分数；
9. 实现 `GCNAdapter`，接入 `cross-contract_detection` 的跨合约结果；
10. 实现推理定位智能体，输出疑似漏洞、定位和解释；
11. 实现修复智能体，输出修复建议和测试建议；
12. 实现 Markdown/HTML 报告生成；
13. 搭建 Web 前端，完成上传、进度、总览、详情和报告展示；
14. 最后再接入真实验证工具和更完整的向量数据库。

### 18.11 最小可演示版本

比赛或阶段性验收建议先完成一个最小可演示版本：

```text
Web 上传 Solidity 源码
→ 后端解析函数
→ LSTM / DeepSVDD / GCN 至少一个模型返回真实或半真实证据
→ JSONL 检索相似案例
→ 风险函数生成函数排行
→ 大模型生成解释与修复建议
→ Web 展示审计报告
```

该版本的重点是展示系统闭环，而不是一次性接满所有模型细节。

### 18.12 工程实现原则

落地时应遵守以下原则：

- 原训练脚本尽量不改，新增 Adapter 包装推理能力；
- Web 系统只加载训练好的模型，不执行训练增强流程；
- `generated_smart_contracts` 只作为训练增强创新点介绍；
- `unknown_detect` 不接入最终系统，避免未知检测入口混乱；
- 所有模型输出必须进入统一 `ModelEvidence`；
- 所有结论必须经过风险函数和状态机，不允许大模型直接改写模型结论；
- JSONL 是知识库事实源，向量库只是召回索引；
- 范围外风险只能作为 warning，不能直接验证和修复；
- 第一版优先保证流程完整、证据清晰、报告可展示。
